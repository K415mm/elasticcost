## 1. Executive Summary
The objective of this analysis is to model and optimize hosting costs for a 9-node distributed data infrastructure. The initial blueprint required **183 GB of RAM** combined with massive storage footprints, notably **2 × 17 TB** on the historical data tier (Frozen nodes). 

By re-engineering the Frozen layer away from a legacy "All Block Storage" model into a modern "Hybrid Object Storage (S3) + Local NVMe Shared Cache" architecture, the global monthly invoice is reduced by **59%**, yielding a net annual savings of **€23,342.40** without compromising Hot tier write/read performance.

## 3. Detailed Target Configuration Breakdown

### 3.1 High-Performance Tier (Hot & Management Layers)
* **3 × hot-node (Master / Data Hot):** Evaluated against the **b3-32** flavor (32 GB RAM, 8 vCores, 200 GB local NVMe included). To fulfill the 1.5 TB target per node, a **1,300 GB Block Storage NVMe** volume is appended to each instance.  
    *Cost Breakdown:* (Basic VM: €112.01) + (Block storage: 1300 GB × €0.0690 = €89.70) = **€201.71 / month per node.**
* **1 × master-tiebreaker (Dedicated Quorum):** Mapped to the **b2-7** flavor (7 GB RAM, 2 vCores, 50 GB local SSD). This flavor completely absorbs the baseline 4 GB RAM / 20 GB storage specification at the absolute lowest cost bracket in the provided catalog.  
    *Cost Breakdown:* **€25.17 / month.**
* **2 × kibana-fleet (Kibana & Fleet HA):** Driven by **b3-16** instances (16 GB RAM, 4 vCores, 100 GB local NVMe). This native storage completely satisfies the required 50 GB local operational disk footprint.  
    *Cost Breakdown:* **€56.00 / month per node.**
* **1 × ml-node (Dedicated Machine Learning Node):** Hosted on a dedicated **b3-16** instance (16 GB RAM, 4 vCores, 100 GB local NVMe) to isolate compute-heavy anomaly detection jobs.  
    *Cost Breakdown:* **€56.00 / month.**

### 3.2 Cost-Effective Storage Tier (Re-engineered Frozen Layer)
The initial setup assigned local Block Storage volumes directly to the compute instances for historical cold data, ballooning costs. The optimized setup leverages **Elasticsearch Searchable Snapshots** over Object Storage:
* **2 × frozen-node (Compute Layer):** Retains the **b3-16** instances (16 GB RAM, 4 vCores, 100 GB local NVMe).
* **Local NVMe Cache Extension:** A dedicated **500 GB Block Storage NVMe** volume is mounted onto each node. Combined with the built-in 100 GB drive, this creates a **600 GB local high-speed cache** (representing an optimal **~3.5%** cache-to-data ratio for regional lookups).
* **Object Mass Storage:** The core **34 TB** data corpus is offloaded entirely to **OVH High Performance Object Storage (S3 API)** at an average operational tier cost of ~€0.011 / GB / month.  
    *Cost Breakdown:* 2 × [ (Compute: €56.00) + (Block Cache: €34.50) + (S3 Storage: 17,000 GB × €0.0110 = €187.00) ] = **€277.50 / month per node.**

## 5. Network Sizing Validation & Core Technical Configurations

Because data on the Frozen tier resides in an Object Storage bucket, system queries fetch indices on-demand. Network pathing must be architected correctly to avoid query starvation.

### 5.1 Network Throughput Analysis
* **Bandwidth Constraint:** The **b3-16** compute instance is limited to a public bandwidth allocation of **1 Gbit/s** (equivalent to a theoretical transfer rate of **125 MB/s** for S3 operations).
* **SLA and Query Impact:** If an analyst issues an exhaustive query targeting historical indices that are not warm in the local cache, pulling a **20 GB data block** from S3 will stream into the local NVMe cache in approximately **2 minutes and 40 seconds**. This fulfills regular SLAs for historical deep-dive analytics.

### 5.2 Critical Cluster Configurations
To prevent network cards from encountering severe packet starvation or triggering node isolation issues during high-volume S3 retrieval states, the following configuration directives must be injected into `elasticsearch.yml`:

```yaml
# 1. Throttling network index recovery traffic to safeguard cluster heartbeat communication
indices.recovery.max_bytes_per_sec: 80mb

# 2. Assigning the custom mounted NVMe block volume for Shared Cache architecture
path.shared_data: /mnt/nvme-cache/

# 3. Defining maximum cache boundaries (leaving a safe 50GB margin for OS operations)
x-pack.searchable.snapshot.shared_cache.size: 550gb
